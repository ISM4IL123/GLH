legal, security, and ethical compliance

GDPR compliance

data transparecy: users must accept terms and conditions befor registering
data minimisation: only name, email and hashed password are collected. no payment card data is stored
right to erasure: not yet implemented, identified as future requirement.
consent: explicit consent obtained for feedback sessions via signed consent forms

WCAG compliance

font resizing: accessibility panel offers large text mode(confirmed functional in the demo)
colour contrast dark theme uses #19106d/#03011d background with #fff text(contrast ratio greater than 7:1) light mode toggles to #f5f5f5 with #000 text
keyboard navigation: full tab-key operability , skip to content link, focus indicators
screen reader: aria lables, aria-live refions for dynamic content, role=”alert” for errors
audit status: manual testing confirmed, automatic lighthouse audit not yet run-added to future tasks.

Priority	Improvement	Source Feedback	Technical Detail
Critical	Migrate to MongoDB/PostgreSQL	Bob:”json causes race conditions”	Add Mongoose ORM, connection pooling, atomic transaction for checkout
Critical	httpOnly cookie authentication	James: “JWT in local storage = XSS risk”	Replace localstorage.setitem(“token”) with res.cookie (“token”,...,{httpOnly: true, secure:true})
Critical	Atomic checkout transaction	Humza:”stock check and write not atomic”	Implement pessimistic locking or MongoDB transactions around stock validation and decrement

Conclusion

the GLH prototype successfully demonstrates functional prodiciency in core marketplace features and meets WCAG 2.1 Level AA accessibility standards, however 3 critical blockers prevent production readiness: the JSON file database, JWT in localStorage and lack of atomic checkout transactions, From a user perspective, producer onboarding visibility and checout simplicity are the highest impact ux improvements. by implementing the version 2.0 plan above GLH can progress from prototype to beta launch within 13 weeks.
