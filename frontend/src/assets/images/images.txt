5. Legal, Security, and Ethical Compliance

5.1 GDPR Compliance

Data transparency: Users must accept Terms and Conditions before registering (simulated via checkbox on signup form).
Data minimisation: Only name, email, and hashed password are collected. No payment card data is stored (client-side validation only).
Right to erasure: Not yet implemented; identified as future requirement.
Consent: Explicit consent obtained for feedback sessions via signed consent forms.

5.2 WCAG Compliance

Font resizing: Accessibility panel offers large text mode (confirmed functional in demo).
Colour contrast: Dark theme uses #19106d/#03011d background with #fff text (contrast ratio greater than 7:1). Light mode toggles to #f5f5f5 with #000 text.
Keyboard navigation: Full Tab-key operability, skip-to-content link, focus indicators.
Screen reader: ARIA labels, aria-live regions for dynamic content, role="alert" for errors.
Audit status: Manual testing confirmed. Automated Lighthouse audit not yet run — added to future tasks.

5.4 Security Measures

- Passwords hashed with bcrypt (salt rounds 10)
- JWT signed with environment secret, 1-hour expiry
- Rate limiting: 300 req/15min global, 30 req/15min auth
- Helmet security headers, CORS restricted, x-powered-by disabled
- Input sanitization strips HTML tags (XSS prevention)
- Zod schema validation prevents malformed data

6.2 Tool Evaluation and Self-Critique

Microsoft Forms: While it allowed easy distribution to 6 testers, analysing 104 technical questions times 3 testers manually was inefficient and error-prone. It lacks automated cross-tabulation, sentiment analysis, and branched logic. For GLH Version 2.0, I will use SurveyMonkey or build a custom SQL-backed survey dashboard to generate automated KPI reports and visualise response distributions instantly.

OBS Studio: Produced high-quality screen recordings but generated 500MB+ files. Future: use lower bitrate or built-in browser recording APIs.

6.3 GLH Version 2.0 — Specific Iteration Plan (Based Only on Feedback)

Priority: Critical
Improvement: Migrate to MongoDB/PostgreSQL
Source Feedback: Roddick: "JSON causes race conditions"
Technical Detail: Add Mongoose ORM, connection pooling, atomic transactions for checkout

Priority: Critical
Improvement: httpOnly cookie authentication
Source Feedback: Callum: "JWT in localStorage = XSS risk"
Technical Detail: Replace localStorage.setItem("token") with res.cookie("token", ..., {httpOnly: true, secure: true})

Priority: Critical
Improvement: Atomic checkout transactions
Source Feedback: Taylor: "Stock check and write not atomic"
Technical Detail: Implement pessimistic locking or MongoDB transactions around stock validation and decrement

Priority: High
Improvement: Add comprehensive test suite

Technical Detail: Jest for unit tests, Supertest for API integration tests, Playwright for E2E checkout journey

Priority: High

Technical Detail: Add address book, Apple Pay / PayPal buttons, one-click checkout for returning users

Priority: High

Priority: High

Source Feedback: Pippa: "Too bland and boring"

6.4 Conclusion

The GLH prototype successfully demonstrates functional proficiency in core marketplace features and meets WCAG 2.1 Level AA accessibility standards. However, three critical blockers prevent production readiness: the JSON file database, JWT in localStorage, and lack of atomic checkout transactions. From a user perspective, producer onboarding visibility and checkout simplicity are the highest-impact UX improvements. By implementing the Version 2.0 plan above, GLH can progress from prototype to beta launch within 12 weeks.
