1.Introduction
 
This task is to gather feedback on my GLH(Greenfield Local Hub) prototype which is a full stack web aplication developed using react as a frontend and an express backend with JSON file based data storage. the website allows users to register and log in with role based access (user,producer,admin), brows and fiter products, manage their shopping cart stored in the local storage and complete a checkout process that generates orders and updates the stock levels.Producers are able to manage their own products through a dedicated dashboard while the admins can review and approve producers aplications.

feedback is an essential stage in the development process because it enables user acceptance testing (UAT) ensuring that the system meets both functional requirements and user expectations before further development. in industry this aligns with iterative and agile methods where software is continuously evaluated and improved based on real user input rather than assumptions made during the initial design by the developers. by exollecting sturctured feedback from technical and non technical users i can identify usability issues, functinal limitations and areas for improvement withing the current iteration of the prototype
this feedback will be used to inform future development (e.g:Task 3b) cycles by highlighting issues such as usability in teh checkout , clarity of the interface and robustness of the producer and admin features ensuring the system evolves in line with the user needs and the original project requirements


2.Feedback Planning

Purpose and objectives
the purpose of this feedback cycle is to evaluate how effective the GLH prototype is through the structured user acceptance testing (UAT) within an iterative development approach. the system includes key functional areas such as user authentication with role based accesss(user,producer,admin) product browsing with category filtering, a local storage based shopping cart , checkout process with order creation , producer/admin dashboards for product/aplication management. the feedback will focus on how well these features meet user needs and identify any usability or functional issues within the current implementation.

to ensure the feedback is measurable and actionable, the following smar objectives have been defined:
•	usability: to determine whether 100% of users can successfully brows products, add items to the cart and complete checkout without any external help
•	design and navigation: to achieve an average usability rating of atleast 80% regarding the clarity of navigating the top bar, product filtering and page structure
•	functional reliability: to identify atleast three functional issues or inconsistencies across key features such as cart persistence, checkout validation or producer dashboard product management
•	role based system effectiveness: to assess whether users clearly understand and can correctly interact with their assigned roles without confusion
•	user experience feedback: to collect atleast 2 good improvement suggestions from each audience type within the feedback session to inform the next iteration
these objectives ensure that both functional performance and user experience are evaluated in a structured and industry aligned manner reflecting the agile development and uat principles

Audience Profiles
technical audience:
the technical audience consists of softwar developers testers or technically skilled users who are able to ecaluate the functionality of the prototype. in this system they would be responsible for reviewing core implementation details such as the express backend routs, json-based data handling, authentication using jwt and client side state management using local storage.
their focus will beo n the reliability, structure and effiecienct of the system.this includes assessing whether api endpoints such as product retrieval checkout processing and producer product management function correctly  under certain conditions. they will also evaluate data consistency between frontend and backend, how stock levels are updated druing checkout and whether role based access controll is correctly enforces across user,producer and admin features.
technical users will also examine security and implementation quality, such as password hashing with bycrypt token based authentication flow and whether the server side validation is sufficient, their feedback will help identify bugs logic errors and areas where the system may not scale or behave predictably

non technical audience:
the non technical audience consists of end users(customers and producers) these users interact with the system without understanding its underlying code or architecture. in the context of this prototyope this includes users browsing and purchasing products as well as producers managing their own listing through the dashboard
their focus will be on usability clarity and the overall user experience for example theiy will evaluate whether the product browsing interface is easy to navigate whether filtering by category is intuitive and if the checkout process is clear and straightforward. they will also assess whether key actions such as adding items to the cart, adjusting quantities and omcpleting an order is easy to understand without help.
producers will specficaly docus on how easy it is to add edit and delete products in teh producer dashboard, while general users will focus on the simplicity of the shopping journey from product selection through to order confirmation. this feedback is essential for identifying usability issues that may not be visible from a technical perspective, such as confusing layouts, unclear button actions or unnecessary complexity in the user journey.


chosen feedback methods
a mixed method approach has been selected to ensure that both quantitive and qualitative data is collected. this is important because the prototype includes functional systems as well as user experience elements.using multiple methods mporoves validity by allowing findings to be cross checked across different types of evidence.




digital questionaire:
a digital questionaire will be used for both technical and non technical usert to ollect structured feedback this method provides data through 1-5 ratings and optional written responses

this method is appropriate because it allows consistend measurement of usability and satisfaction across all users. for example users can rate how easy it is to navicate between pages, clarity of checkout process and overall ease of completing a purchase
this data directly supports evaluation of key system features such as the homepage, product filtering, cartpage functionality and checkoutpage. it also allows comparison of responses across different users to identify common usability issues
